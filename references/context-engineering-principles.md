# Context Engineering Principles

에이전트 역할 분리, 문서 계층화(Overview/Detail), 스킬의 조건부 로딩이 왜 필요한지에 대한 이론적 근거를 정리한다. 프롬프트 엔지니어링이 컨텍스트 엔지니어링으로 진화한 흐름의 핵심을 담는다.

Related: agent-architecture (base architecture doc), [Elvis Sun / Zoe Orchestrator Pattern](./elvis-zoe-orchestrator-pattern.md)

---

## Detail

### 핵심 전제: 컨텍스트 윈도우는 제로섬

LLM의 컨텍스트 윈도우는 유한하다. 200K든 1M이든, 창 안에 넣는 정보가 많을수록 **신호 대 잡음비(SNR)**가 떨어지고 출력 품질이 측정 가능하게 하락한다. 단순히 "들어가는 양"이 아니라 "관련 있는 정보의 비율"이 품질을 결정한다.

### 3대 원칙

#### 1. 컨텍스트 순도 (Context Purity)

각 에이전트가 보는 정보의 관련성을 최대화한다.

| 적용 방식 | 효과 |
|----------|------|
| 문서를 Overview/Detail로 분리 | 오케스트레이터는 개요만 보고 판단 |
| 에이전트마다 보는 범위 제한 | SNR 향상 → 출력 정확도 상승 |

#### 2. 인지적 책임 분리 (Cognitive Separation)

비즈니스 사고와 코드 실행은 완전히 다른 인지 모드다. 한 에이전트가 둘을 동시에 하면 모드 전환 비용이 발생하고 양쪽 품질이 모두 떨어진다.

```
Layer 1 (비즈니스)    "왜 만드는가, 누가 쓰는가"
    ↓ 계획 전달
Layer 2 (오케스트레이션) "어떻게 분해하고, 누가 수행하는가"
    ↓ 태스크 분배
Layer 3 (실행)        "이 파일의 이 함수를 이렇게 고친다"
```

각 레이어는 상위 레이어의 컨텍스트를 오염시키지 않는다.

#### 3. 토큰 경제성 (Token Economy)

같은 토큰 예산으로 더 깊은 작업을 할 수 있게 한다.

| 적용 방식 | 절약 효과 |
|----------|----------|
| 스킬은 필요 시에만 로드 | 12개 × 300줄 = 3,600줄 상시 절약 |
| 에이전트별 모델 차등 배정 | 단순 작업에 고비용 모델 낭비 방지 |

### 이 원칙의 출처

- **Elvis Sun / Zoe 패턴** — "컨텍스트 창은 제로섬" 관찰에서 2-Tier 분리 도출
- **Anthropic 공식 가이드** — 긴 메가 프롬프트 대신 작은 skill artifact로 쪼개고 필요 시 로드
- **업계 흐름** — 프롬프트 엔지니어링 → 컨텍스트 엔지니어링으로의 전환

### 한 줄 요약

> "LLM의 주의력은 유한하다. 각 에이전트가 그 순간 꼭 필요한 맥락만 갖고 일하게 만들어야 최고 품질이 나온다."
